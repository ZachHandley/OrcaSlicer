# CLAUDE.md

OrcaSlicer — open-source C++17 3D slicer. wxWidgets GUI, CMake build system.

## Build Commands

```bash
# macOS
cmake --build build/arm64 --config RelWithDebInfo --target all --

# Linux
cmake --build build --config RelWithDebInfo --target all --

# Windows (replace %build_type% with Debug/Release/RelWithDebInfo)
cmake --build . --config %build_type% --target ALL_BUILD -- -m
```

## Testing

Catch2 framework. Tests in `tests/` directory.

```bash
cd build && ctest --output-on-failure           # all tests
ctest --test-dir ./tests/libslic3r              # individual suite
ctest --test-dir ./tests/fff_print
```

## Code Style

- C++17, selective C++20. PascalCase classes, snake_case functions/variables
- `#pragma once` for headers. Smart pointers and RAII preferred
- Parallelization via TBB — be mindful of shared state

## Key Entry Points

- App startup: `src/OrcaSlicer.cpp`
- Slicing pipeline: `src/libslic3r/Print.cpp`
- All print/printer/material settings: `src/libslic3r/PrintConfig.cpp`
- GUI: `src/slic3r/GUI/`
- Core algorithms: `src/libslic3r/` (GCode/, Fill/, Support/, Geometry/, Format/, Arachne/)
- Printer profiles: `resources/profiles/[manufacturer].json`

## Engine boundary (Phase 0+)

`libslic3r` ships as the shared library `liborca-engine.so` with a wx-free public API. Out-of-process / FFI / plugin consumers drive it through this boundary, never through `src/slic3r/GUI/`.

- **C++ public headers**: `engine/include/orca/{Session,Presets,Project,Slicer,Export,Events,EventTypes,Config,ConfigKeys,Result,Globals}.hpp`. Entry point: `orca::Session::create()` (returns `std::unique_ptr<Session>`); access services via `session->{presets,project,slicer,exporter,events}()`. Errors propagate as `orca::Result<T>` (`Result.hpp`).
- **Typed config surface**: compile-time-keyed via the generated `orca::keys::*` tags in `engine/include/orca/ConfigKeys.hpp` (regenerated by `engine/scripts/catalog/regen_keys.sh`). Reads/writes go through `orca::config::get<K>/put<K>` (adhoc receiver) or `Presets::get<K>(Scope)` (scoped). Runtime-string-key access still goes through `ConfigBase::option(key)` — used by the C ABI.
- **C ABI**: `engine/include/orca/c_api.h` (POD-only, opaque-handle); implementation in `engine/src/c_api.cpp` (35 `orca_*` exports). All C++ exceptions are caught at the boundary; errors flow through `orca_error_code_t` + a thread-local `orca_last_error_message()`. The events bridge maps all 8 `orca_event_kind_t` values to typed `Events::subscribe<E>` registrations.
- **Boundary invariant**: `engine/scripts/verify_no_wx.sh [path-to-liborca-engine.so]` fails non-zero on any wxWidgets symbol. CI runs it on every Linux build (`build_orca.yml`). When passing a path, prefer `build/src/libslic3r/RelWithDebInfo/liborca-engine.so` — without an explicit path the script can pick a stale `Release/` copy.
- **Engine tests**: pure-C ABI smoke at `tests/engine-c-abi/` (ctest `engine_c_abi`); Catch2 v3 suite at `tests/engine/` (ctest `engine_tests`). Headless slice/export canary at `engine/cli/orca-engine-cli` — also asserts the event bus delivers `SlicingProgress`/`SlicingFinished`/`ExportFinished` during the service slice.
- **GUI publishes additively**: `BackgroundSlicingProcess` + `PartPlate` publish typed lifecycle events on the engine bus alongside (never replacing) the existing wx events, so plugin consumers see GUI-triggered slices too. Handle `0` is the "interactive GUI slice" sentinel; engine-issued handles start at `1`.

## Critical Constraints

- **Backward compatibility required** for .3mf project files and printer profiles
- **Cross-platform** — all changes must work on Windows, macOS, and Linux
- Profile/format changes need version migration handling
- Dependencies built separately in `deps/build/`, then linked to main app
- **Engine remains wx-free** — anything under `engine/include/orca/` or `engine/src/` MUST NOT include wxWidgets. `engine/scripts/verify_no_wx.sh` runs in CI and will fail the build.

## Code review focus areas

- Changes must not cause regressions in existing functionality, defaults, profiles, or project compatibility.
- Features gated by options must not affect existing behavior when those options are disabled.
- Changes should follow the existing code style and architecture. Architectural changes should be justified in code comments and the PR description.
- Add helper functions or utilities only when existing code cannot reasonably be reused. Avoid duplication.
- Keep code concise and clear. Manually simplify AI generated bloated codes before review.
- Include targeted tests or documented verification for behavior changes, especially in slicing logic, profiles, formats, and GUI defaults.
